💡 1. What this app is

This is a trading bot for NIFTY options — a program that watches the live NIFTY price and, following a set of rules, decides when to buy and sell options. What makes it a little unusual is that it runs several different trading strategies at the same time, all fed by one shared live price feed. So instead of running four separate programs, one program listens to the market once and hands that same price to every strategy.

Each strategy can run in three ways: practice (real prices, pretend money), real money (actual orders through your broker), and testing on the past (see how it would have done on old data). There's a web dashboard you open in a browser to start and stop everything, watch trades happen, and change settings — you never need to touch the code to use it day to day.

How it's all wired — one picture

Here's the whole shape of the app. One live price line comes in from Fyers. A single sharing hub (the "socket manager") takes each price tick once and hands the very same tick to every strategy — that's the "fan-out". Each of the six strategies can then run in any of its three ways at once.

Why one shared line matters. Opening a second price line to the broker is not allowed — every strategy listens through the one hub. That's what lets six strategies run side by side on a single connection without stepping on each other.
You don't need to be a programmer to run it. This guide walks through the setup one command at a time and explains, in plain words, what each command does and why you're running it. If a word looks technical, check the next section — it's probably explained there.

📚 2. Words you'll see (plain meanings)

A handful of technical words come up during setup. Here's what each one really means, in everyday language. Skim it once and keep it handy — the rest of the guide reads easily after this.

WordWhat it actually means
Node.jsThe engine that runs this app. The app is written in a language called JavaScript, and Node.js is the program that reads that code and makes it work. You install it once on the computer.
npm"Node Package Manager" — a helper that comes with Node.js. It downloads the ready-made building blocks (other people's code) that this app is built on top of. Think of it as the app's shopping assistant.
nodemonA convenience tool for development. It runs the app and, every time you edit a file, automatically restarts it for you so you see your change straight away. You only use it while tinkering, not in real running.
PM2A "keep it running" manager. On the real server, PM2 launches the app, restarts it if it ever crashes, and starts it back up automatically after a reboot — so the bot stays alive without you babysitting it.
.env file (environment file)A plain text file named .env that holds your settings and secret keys (broker passwords, phone/Telegram details, on/off switches). The app reads it on startup. It's kept out of the shared code so your secrets stay private.
OAuth token / login tokenA temporary "pass" the broker gives the app after you log in, proving it's allowed to see prices and place orders on your behalf. Like a day-pass wristband — it works for a while, then expires and you log in again.
WebSocketA always-open phone line between the app and the broker that streams live prices non-stop, the instant they change. This app opens exactly one such line and shares it with every strategy.
EC2A rented computer in Amazon's cloud that stays on 24/7. The bot lives there so it can trade during market hours even when your own laptop is off. "The server" and "EC2" mean the same thing here.
HTTPS / SSL certificateHTTPS is the secure (padlock) version of a web address. To use it, the app needs an "SSL certificate" — a small identity file. We make a self-signed one, which works fine but makes the browser show a one-time "are you sure?" warning (see Section 11).
Fyers / ZerodhaThe two stock brokers this app connects to. Fyers supplies the live prices (and places orders for some strategies); Zerodha places orders for others. You need an account with each.

✅ 3. What you need before you start

  • A computer to run it on. For real trading this is a cloud server (EC2) that stays on all day. For trying things out, your own laptop is fine.
  • Node.js version 16. This is important — the app is pinned to Node 16 on the server. Newer versions (18, 20) will not run there, because the server's operating system (Amazon Linux 2) is too old for them. On the server, Node 16 is selected automatically during deployment; on your own laptop, install Node 16 yourself.
  • The app's code. A copy of this project folder on the machine.
  • Broker accounts. A Fyers account (for prices and some orders) and a Zerodha account (for other orders), each with its "developer/API" keys set up.
  • Optional: a Telegram bot. If you want the app to text you when it starts, enters, or exits a trade, you set up a Telegram bot and paste its details into the settings. This is optional.
Why Node 16 exactly? The cloud server runs an older system that only supports up to Node 16. Node 18 and 20 need a newer system component the server doesn't have, so they simply won't start there. When in doubt, use Node 16.

🚀 4. Install & run — step by step

Follow these in order. Each step has the exact command to type (in the dark box) and, underneath, a plain note on what it does and when you run it. Every command is typed into a "terminal" — the text window where you talk to the computer.

  1. Get the code onto the machine. 
    git clone <repo> trading-bot
cd trading-bot
    This downloads the project into a folder called trading-bot and then steps inside it. You only do this once. (If the folder is already there, skip this.)
  2. Download the building blocks. 
    npm install
    This fetches all the ready-made pieces of code the app is built on (the price-feed library, the broker libraries, and so on). Run it once after you first get the code, and again only when the building blocks change.
  3. Create your settings file. 
    cp .env.example .env
    This makes your own settings file (.env) by copying the example one. Then open .env in any text editor and fill in your details — your Fyers keys, your Zerodha keys, your server address, and (optionally) your Telegram details. This file holds your secrets, so keep it private.
  4. Make the security certificate (one time). 
    mkdir -p certs
openssl req -x509 -newkey rsa:4096 \
  -keyout certs/key.pem -out certs/cert.pem \
  -days 3650 -nodes -subj "/CN=$(curl -s ifconfig.me)"
    The app serves its dashboard over the secure (padlock) HTTPS address, and for that it needs a certificate. This command creates a self-signed one that's good for ten years. You do this once. (Without it the app refuses to start — see Section 11.)
  5. Start the app. On the real server, start it with the keep-it-running manager, PM2: 
    pm2 start src/app.js --name trade
    This launches the bot and names it trade so PM2 can watch over it. If it ever crashes, PM2 brings it straight back.
  6. Make it survive a reboot. 
    pm2 startup && pm2 save
    This tells the server to start the bot automatically the next time it's switched on or restarted — so you don't have to log in and start it by hand. One-time step.
  7. Open it in your browser. Go to your server's address on port 3000, using https: 
    https://YOUR_SERVER_IP:3000
    That's the dashboard. From here on you start, stop, and configure everything by clicking, not by typing commands. (See Section 10 for the first-visit browser warning.)
Just testing on your own laptop? Steps 1–4 are the same. For step 5, instead of PM2 you can run npm run dev — that starts the app and auto-restarts it whenever you change a file, which is handy while experimenting. Then open https://localhost:3000. Use PM2 (steps 5–6) for the real, always-on server.

⚙ 5. The commands, explained

These are the handful of commands you'll actually use. You don't need to memorise them — just know what each one is for.

CommandWhat it does — and when you run it
npm install Downloads all the building blocks the app needs. Run it once after you first get the code (and again only if the building blocks are updated).
npm run dev Starts the app in "development" mode using nodemon, which auto-restarts it every time you edit a file. Great while tinkering on your own machine; you'll see changes without stopping and starting by hand.
npm start Starts the app plainly, with no auto-restart. It's the same as running node src/app.js. A simple way to run it once without PM2.
node -c src/app.js A quick spell-check for the code — it reads the main file and confirms there are no obvious typos that would stop it starting, without actually running it. Fast way to sanity-check before deploying. (There's no separate test or build step in this project.)
pm2 startOrRestart ecosystem.config.js --update-env The production reload. It (re)starts the app under PM2 using the project's settings file (ecosystem.config.js), picking up the latest memory limits and settings. This is what the automatic deployment runs on the server after new code arrives.
pm2 start src/app.js --name trade First-time launch under PM2, naming the app trade. After this, PM2 keeps it alive and restarts it if it crashes.
pm2 startup && pm2 save Tells the server to start the app on its own after any reboot. One-time step.
How new code reaches the server. When approved changes are saved to the project's main branch, the server picks them up automatically: it selects Node 16, installs any new building blocks, and reloads the app with pm2 startOrRestart ecosystem.config.js --update-env. You usually don't run these by hand — it happens for you.
A reload won't strand an open real trade. When the app is stopped for a deploy or restart, it now closes out any real (live) positions across all strategies first, and it's careful not to cut the shared price feed while another strategy still has a live trade running on it. So a routine restart no longer risks leaving a real position open and unwatched.

🔑 6. Logging the brokers in (tokens)

The app can't see prices or place orders until you log the brokers in. You do this from the app's own login page in the browser — it sends you to Fyers (and Zerodha), you sign in there, and each broker hands back a login token (a temporary pass). The app saves those passes and uses them for the rest of the session.

BrokerWhat it's forWhere the pass is saved
FyersThe live price feed for every strategy, plus real orders for some of them.~/trading-data/.fyers_token
ZerodhaReal orders for the other strategies.~/trading-data/.zerodha_token
Most common gotcha: the Fyers pass expires daily. The Fyers login token stops working after about a day. When it's expired, running a backtest (testing on past data) comes back with "no historical candles" or 0 candles — it looks like a data problem, but it almost always just means the Fyers token has expired. The fix is simple: log in to Fyers again from the app, and backtests work immediately. It does not mean the dates are wrong or the strategy is broken.

Because these passes are saved under ~/trading-data/ (see the next section), they survive code updates and restarts — you only log in again when a pass actually expires.

💾 7. Where your data is kept

Everything the app needs to remember — your trade history, your broker passes, your recorded price data — lives in a folder called ~/trading-data/, which sits outside the app's code folder. That's on purpose and it's important:

Your history is safe across updates. Because the data lives outside the code folder, pulling new code or restarting the app never wipes your trades or your broker logins. New versions of the app come and go; your data stays put.

Here's what's in that folder, in plain terms — you rarely need to open any of it, but it helps to know it's there:

What's in itIn plain words
.fyers_token / .zerodha_tokenYour broker login passes (see Section 6).
*_paper_trades.json / *_live_trades.jsonYour saved trades — separate files for each strategy, and separate ones for practice ("paper") vs real money ("live").
.active_*_position.jsonA safety snapshot of any open trade, so that if the app restarts mid-trade it can pick the trade back up. Now kept for all six strategies (EMA_RSI_ST, BB_RSI, PA, EMA9_VWAP, ORB and Trend_PB). On restart the app checks each snapshot against the broker and only clears it when the broker's list of positions is provably readable; if the broker pass has expired (an empty, unreadable list), the snapshot is kept and you're warned — so a real open trade is never quietly forgotten.
trades/ and *_log.jsonlA day-by-day written record of every trade — the permanent audit trail.

backtest_cache/ / candle_cache/Saved copies of past price data so testing is faster. The app cleans these up automatically.

🔧 8. Changing settings

You don't edit files to change how the app behaves — you use the Settings page inside the app (there's a "Settings" link in the menu). Every switch and number you can change there — which strategies are on, how much it's allowed to lose in a day, what time it stops trading, and so on — is saved and used the next time a strategy runs.

  • Each strategy has its own pages. Every strategy gives you three: a Paper page (practice), a Live page (real money), and a Backtest page (test on the past). You open the one you want from the menu.
  • Settings are just labelled switches. Behind the scenes each setting is a named on/off switch or number (what programmers call an "environment variable"), but on the Settings page they're shown as plain toggles and boxes you fill in.
  • Every change is remembered with your notes. When you save the Settings page it can tag the change with a one-line note, and it writes a snapshot of the settings into that day's trade record — so you always know exactly which settings produced which trades.

🎯 9. Practice, Real money & Testing

Every strategy can run in three ways. It's worth understanding the difference before you ever risk real money.

ModeWhat happens
Paper (practice)Real live prices, but pretend money — no real orders are ever placed. This is the safe way to watch a strategy work and the "source of truth" everything else is checked against.
Live (real money)Real orders through your broker. Locked behind two safety switches (below) so it can't fire by accident.
Backtest (test on the past)Runs the strategy over old days to see how it would have done. Needs a valid Fyers login for the historical prices (see Section 6).

What happens for one trade, start to finish

Whichever mode a strategy is in, every trade travels the same path. Live places a real order at the "order" step; practice and testing only pretend at that step — everything before and after is identical.

Tick receiveda new live price arrives → Signal checkdo the strategy's rules say "enter"? → Risk & guardsVIX, bid-ask spread, daily-loss, whole-account cap → Order placedreal order — or dry-run just logged → Position monitoredstop-loss, trail, time-stop watched each tick → Exita rule fires and closes the trade → Saved + Telegramtrade written to history, you're messaged
The guards can stop a trade before it ever opens. If the market is too jumpy (VIX), the option's buy/sell gap is too wide, the day's loss limit is hit, or the whole-account cap has tripped, the "Risk & guards" step quietly blocks the entry — no order is placed and the strategy waits for the next chance.

The two safety switches on real-money trading

Placing a real order is deliberately hard to do by accident. Two separate switches must both be set the right way before a single real order can go out:

1. The strategy's own live switch (for example ORB_LIVE_ENABLED) must be turned on, and 2. the global "dry-run" switch LIVE_HARNESS_DRY_RUN must be turned off. By default the dry-run switch is on, which means even a "Live" page only pretends: it writes down the order it would have placed but sends nothing to the broker. A fresh install therefore cannot place a real trade until you deliberately turn dry-run off.
{STRATEGY}_LIVE_ENABLEDthis strategy's own live switch is ON AND LIVE_HARNESS_DRY_RUN=falsethe global dry-run switch is OFF AND per-strategy dry-run OFFwhere a strategy has its own override → Real order goes to the broker

If any gate is not set the right way → the app only writes down the order it would have placed (a dry-run log) and sends nothing.

The safe habit: prove a strategy behaves the same in practice and in "pretend live" first — only then turn the dry-run switch off for real trading.

Extra safety nets that protect the whole account

Beyond the two switches, a few newer guards sit above the individual strategies to protect the account as a whole. All of them are safe by default:

  • A whole-account daily loss cap. Each strategy already limits how much it can lose in a day — but if every strategy hit its own limit on the same bad day, the combined loss could be much bigger. The new cap (PORTFOLIO_MAX_DAILY_LOSS) adds up today's real profit-and-loss across all six strategies and, once the combined loss reaches your figure, quietly stops every strategy from opening new trades for the rest of the day. It only ever blocks a new entry — it never places or changes an order — and if it can't read the numbers for any reason it lets trading continue rather than freeze the whole book.
  • An exchange-side "disaster stop" for real trading. When turned on (HARNESS_EXCHANGE_SL_ENABLED), each real trade also leaves a backup stop-loss parked at the exchange, set as a percentage of the entry price (HARNESS_SL_PCT). So even if the app crashes mid-trade, the exchange itself can still close the position. The app's own moment-to-moment stop stays the main one; this is just a safety net, and it's cancelled before any normal exit so it can't fire twice.
  • A "fat-finger" size limit. If someone accidentally sets the trade-size multiplier far too high (say 50×), a ceiling (MAX_LOT_MULTIPLIER, default 10) clamps it back down — and this now covers the manual "enter a trade" buttons too, not just the automatic entries.
These extra guards are OFF or set to safe defaults out of the box. The whole-account cap is 0 (off) until you choose a figure, and the exchange-side disaster stop is off until you switch it on — and because it places a real resting order, prove it on a pretend-live day first. The fat-finger limit is always on at a sensible default.
Backtests now behave more like real life. Testing on the past was made more honest platform-wide: it uses the previous day's volatility (VIX) reading instead of peeking at the day it's testing, it fills stops fairly when the price gaps straight through them, and it assumes realistic slippage. So backtest results are a little more conservative — and closer to what live trading actually delivers.

🌐 10. Opening it & finding the guides

  1. Open the dashboard in a browser. Type your server's address with https and port 3000: 
    https://YOUR_SERVER_IP:3000
    On your own laptop that's https://localhost:3000.
  2. Accept the one-time security warning. Because the certificate is self-signed (Section 2), the browser shows a "your connection is not private" page the first time. That's expected — click Advanced, then Proceed / Continue. It's your own server, so it's safe.
  3. Log the brokers in. From the dashboard, use the login page to sign in to Fyers (and Zerodha). Now the app can see prices and, once you allow it, trade.
  4. Read the strategy guides on the Docs page. There's a Docs link in the menu (or go to /docs). It lists the plain-English guides for each strategy — for example the Opening Range Breakout (ORB) guide — kept alongside this setup guide.
The Docs page is just this folder, shown in the browser. Any guide placed in the app's documents/ folder shows up on the /docs page automatically, so that's the one place to look for how each strategy works.
New: a day-by-day Consolidation Report. On the Edge Analytics page there's now a 📑 Consolidation Report button. It opens one table with a row per trading day — each strategy's trades and profit-or-loss, then the day's totals, wins, losses and net — for practice and real-money trades together. You can filter it by date range (this week, last month, a custom span, and so on) and Save it as a PDF straight from the browser. It only reads your existing trade history; it never changes anything. You can hide the button from the Settings page (it's shown by default).

⚠ 11. When something goes wrong

What you seeWhat it usually means & the fix
A backtest returns "0 candles" or "no historical candles" Almost always an expired Fyers login, not a data or date problem. Log in to Fyers again from the app and re-run — it works immediately.
The browser warns "your connection is not private" Normal on first visit — the certificate is self-signed. Click Advanced then Proceed. It's your own server.
The app won't start / keeps stopping Usually the security certificate is missing or the settings file has a bad value. Make sure you did the certificate step (Section 4, step 4) and that your .env is filled in. The app deliberately stops itself (rather than restarting forever) when the certificate or settings are wrong, so PM2 doesn't loop.
A "Live" page isn't placing real orders Expected unless you've thrown both safety switches (Section 9): the strategy's own live switch on, and the global dry-run switch off. Until then, Live only pretends.
Can't reach the dashboard at all Check the address uses https and port 3000, and that the server's firewall allows port 3000. On the server, pm2 status shows whether the app is running.
New code didn't take effect on the server Make sure the reload used the settings file: pm2 startOrRestart ecosystem.config.js --update-env. Reloading without naming that file reuses the old settings.
